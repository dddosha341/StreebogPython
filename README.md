# Стрибог (ГОСТ 34.11-2018) — Python

Лабораторная работа: **Парадокс дней рождения и криптографические хэш-функции**.

Полноценная реализация хэш-функции Стрибог по ГОСТ 34.11-2018, включая

## Теория

### Стрибог

**Стрибог** (Streebog, ГОСТ Р 34.11-2012 / ГОСТ 34.11-2018) — российская
криптографическая хэш-функция, стандартизированная как национальный стандарт.

- Размер блока: 512 бит (64 байта)
- Выходной размер: 512 или 256 бит
- Использует конструкцию Миягучи-Пренеля
- 12 раундов подстановочно-перестановочной сети (SPN)
- Каждый раунд: S (нелинейная подстановка) → P (перестановка) → L (линейное преобразование)

### Усечение до 48 бит (MSB48)

`h48(x) = MSB48(STREEBOG-512(x))` — берём первые 6 байтов 512-битного хэша.
Это **не** стандартная хэш-функция, а учебная усечённая версия для демонстрации
birthday attack.

### Почему Birthday Attack применим

По парадоксу дней рождения, для n-битного хэша коллизия ожидается после
≈ 1.25 × 2^(n/2) попыток. Для n=48: ≈ 2^24 ≈ 16.7 млн попыток — вычислительно
достижимо за минуты-часы.

### Почему коллизия MSB48 ≠ коллизия полного хэша

Коллизия для MSB48 означает совпадение только первых 48 бит из 512.
Полные 512-битные хэши различаются. Нахождение коллизии для полного
Streebog-512 потребовало бы ≈ 2^256 попыток — вычислительно невозможно.

## Ускорение через T-таблицы

### Базовая реализация

Каждый раунд выполняет LPS = L(P(S(x))):
- **S**: 64 подстановки через S-box (PI)
- **P**: перестановка 64 байтов по таблице TAU
- **L**: 8 групп × 64 бит × перебор 64 строк GF(2)-матрицы = ~512 условных XOR

### Оптимизированная реализация

8 предвычисленных таблиц `T[k]`, k = 0..7, по 256 записей × 8 байт.

```
T[k][v] = XOR of A[k*8 + bit]  для каждого установленного бита в PI[v]
```

Полный LPS для одной L-группы:

```
result_g = T[0][x[g]] ^ T[1][x[8+g]] ^ T[2][x[16+g]] ^ ... ^ T[7][x[56+g]]
```

64 табличные подстановки + 64 XOR-а вместо 64 S-box + перестановки + 512 условных XOR.

### Доказательство эквивалентности

T-таблица объединяет эффект S, P и L для каждого входного байта.
Поскольку L — линейное преобразование над GF(2), вклад каждого байта
может быть вычислен независимо и сложен (XOR). Результат математически
идентичен последовательному применению S → P → L.

## Алгоритм поиска коллизии

1. Генерируем случайное 16-байтовое сообщение `x`
2. Вычисляем `h48(x) = Streebog-512(x)[:6]`
3. Ищем `h48(x)` в словаре `seen`
4. Если найдено и сообщения различны → коллизия
5. Если нет → добавляем `(h48 → x)` в `seen`
6. Повторяем

Ожидаемое число попыток: ≈ 2^24 ≈ 16.7M. Память: ~800 МБ (dict с 16M записями).

## Осмысленная коллизия

### Подход

1. Создаём два BMP-изображения 64×64 (паттерны "X" и "O")
2. Дописываем случайные суффиксы после пиксельных данных
3. BMP-просмотрщики игнорируют данные за пределами объявленного размера
4. Birthday attack между множествами суффиксов для A и B

### Оптимизация

Предвычисляем состояние хэшера после всех полных блоков изображения.
Для каждого суффикса клонируем состояние и дообрабатываем только хвост.
Это сокращает вычисления с O(размер изображения) до O(размер суффикса).

### Результат

- `image1_X.bmp` — белые диагонали на чёрном фоне
- `image2_O.bmp` — белое кольцо на чёрном фоне
- Оба файла открываются стандартными просмотрщиками
- `MSB48(Streebog-512(image1)) == MSB48(Streebog-512(image2))`

## Порядок байтов (endianness)

Стрибог использует **little-endian** внутреннее представление:
- Байт `[0]` — младший (LSB), байт `[63]` — старший (MSB)
- Арифметика (счётчик N, сумма Σ) — LE
- Матрица L: каждая 8-байтовая группа = 64-бит LE-слово

RFC 6986 отображает значения как big-endian числа (MSB слева).
Наша реализация возвращает байты в LE-порядке (как gostcrypto).

Для сравнения с RFC-значениями: `digest[::-1].hex()`.

## Ограничения и допущения

1. **Чистый Python** — не используются C-расширения, производительность ограничена
2. **Коллизия MSB48** — поиск занимает ~30-60 минут в зависимости от машины
3. **Осмысленная коллизия** может потребовать значительной памяти (~800 МБ)
4. **BMP-суффиксы** — некоторые строгие валидаторы могут отклонить файлы с данными
   после объявленного размера, но стандартные просмотрщики (Preview, GIMP, браузеры)
   работают корректно
5. **A-матрица** — значения приведены в LE-нумерации битов (отличается от порядка
   записи в тексте RFC 6986)
6. **Константы C[1..12]** — хранятся в LE-порядке (развёрнуты относительно RFC)
